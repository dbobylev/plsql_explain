# plsql_explain

Инструмент для глубокого анализа PL/SQL кода Oracle. Позволяет получить понятное описание логики любого метода с учётом всех транзитивных зависимостей.

## Задача

Разработчик вводит имя метода — система возвращает подробное текстовое описание его работы, включая все вложенные вызовы, обращения к таблицам и зависимые пакеты. Цель — быстрее разбираться в сложной legacy кодовой базе на PL/SQL без необходимости вручную трассировать зависимости.

## Пример использования

**Вход:**
```
pkg_orders.calculate_total
```

**Выход:** текстовый файл `pkg_orders.calculate_total.txt` с подробным описанием вида:

```
1. Метод pkg_orders.calculate_total
   Назначение: вычисляет итоговую сумму заказа с учётом скидок и налогов.

   1.1. Вызов pkg_discount.get_rate
        Получает процент скидки для клиента по его категории.

   1.2. Вызов pkg_tax.compute
        Рассчитывает сумму налога.

        1.2.1. Вызов pkg_reference.get_code
               Получает код налоговой ставки из справочника.

   1.3. Условие IF total > threshold
        Если итоговая сумма превышает порог — применяется дополнительная скидка (п. 1.1).

   1.4. Вызов pkg_audit.log_operation
        Записывает факт расчёта в журнал аудита.
```

## Архитектура

- **Иерархическая суммаризация снизу вверх** — для обработки больших объёмов кода без переполнения контекста LLM

## Как это работает

### 1. Получение исходного кода

Python-сервис подключается к Oracle БД и извлекает исходный код объектов через `DBA_SOURCE`. Исходники сохраняются в SQLite для последующей обработки.

> На первом этапе источник — только Oracle БД. В перспективе планируется поддержка файлов из git-репозитория.

### 2. Препроцессинг

ANTLR4 парсит исходный код каждого объекта и строит граф:
- дерево вызовов (какой метод что вызывает)
- обращения к таблицам (чтение/запись)
- зависимости между пакетами

Для каждого объекта вычисляется хэш исходника — повторная обработка запускается только при изменении кода. Результат сохраняется в SQLite.

### 3. Сбор дерева зависимостей

По имени метода система обходит граф в глубину и собирает все транзитивные зависимости:

```
pkg_orders.calculate_total
├── pkg_discount.get_rate        (leaf)
├── pkg_tax.compute
│   └── pkg_reference.get_code  (leaf)
└── pkg_audit.log_operation      (leaf)
```

Циклические зависимости обнаруживаются и обрываются: вместо повторного анализа подставляется заглушка со ссылкой на уже описанный метод.

### 4. Иерархическая суммаризация (снизу вверх)

Ключевой механизм для работы с большим и сложным кодом:

1. Суммаризируются листовые узлы (методы без внешних зависимостей)
2. При анализе родительского метода вызовы уже суммаризированных методов заменяются их кратким описанием
3. Процесс поднимается вверх до целевого метода

Это позволяет анализировать глубокие деревья зависимостей, не превышая контекстное окно модели.

### Обработка сложных конструкций

| Конструкция | Подход |
|---|---|
| `IF-ELSE` блоки на 1000+ строк | AST-чанкинг по веткам, суммаризация каждой ветки отдельно |
| SQL-запросы на 500+ строк | Отдельная суммаризация: "что читает/пишет этот запрос" |
| Внешние системные пакеты (UTL_FILE, DBMS_OUTPUT и др.) | Подставляется понятное описание назначения без анализа внутренностей |
| Зашифрованные (WRAPPED) пакеты | Не рассматриваются (отсутствуют в целевой кодовой базе) |
| Ссылки на пакеты | Граф собирает только реально вызываемые объекты |
| Кодовая база 5M+ строк | Обрабатывается только поддерево целевого метода |

## Хранение данных

```
SQLite:
  ├── object_source    — исходный код объектов + хэш для инкрементальной индексации
  ├── call_graph       — граф вызовов между объектами
  └── object_summary   — кэш суммаризаций (не пересчитываем при неизменном исходнике)
```

## Компоненты

- **Сервис получения исходников** (Python) — подключение к Oracle БД, извлечение кода через `DBA_SOURCE`, сохранение в SQLite
- **ANTLR4 сервис** (C# console app) — парсинг PL/SQL, построение графа зависимостей
- **Сервис индексации** (Python) — запуск ANTLR4, инкрементальное обновление по хэшу
- **Сервис анализа** (Python) — иерархическая суммаризация, запись итогового описания в файл
- **SQLite** — хранение исходников, графа и кэша суммаризаций
- **LLM** — корпоративная модель с GPU, доступна через OpenAI-совместимый API

## Технологии

- Python
- ANTLR4 (C#, готовая грамматика PL/SQL с возможностью последующей доработки)
- SQLite
- Gitea CI/CD

## Ограничения

- Production-развёртывание на хосте Windows без Docker, в закрытой корпоративной сети
- На первом этапе источник исходников — только Oracle БД (`DBA_SOURCE`)
- Зашифрованные (WRAPPED) пакеты не анализируются

## План реализации

1. **Сервис получения исходников** — подключение к Oracle БД, выгрузка через `DBA_SOURCE` в SQLite
2. **ANTLR4** — парсинг PL/SQL → построение графа вызовов и таблиц
3. **Сервис индексации** — сохранение графа и исходников в SQLite, инкрементальное обновление по хэшу
4. **Обход дерева зависимостей** по имени метода (без LLM)
5. **Суммаризация листовых узлов** через LLM
6. **Иерархическая сборка** итогового описания → запись в текстовый файл
